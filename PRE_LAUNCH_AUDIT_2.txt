FlowMonix — Pre-Launch Audit Report
Date: 2026-04-19
Auditor: Full codebase review
====================================


CRITICAL ISSUES — Must fix before launch
==========================================

C1. Alert creation has zero plan limit enforcement
    File: src/app/api/alerts/route.ts — POST handler

    The route inserts an alert rule with no check against how many alerts the
    org already has. Any free-plan user can bypass the 2-alert limit by calling
    the API directly. The instance creation route (POST /api/instances) does
    this correctly — getPlanLimits + count check — but alerts don't.

    Also: there is no check that channel === "slack" requires a Pro/Team plan.
    A free user can create a Slack alert via the API.

    Fix: Before the DB insert, fetch org plan + current alert count. Return 403
    if at limit. Block slack channel for free plan.

---

C2. Member invites have zero plan limit enforcement
    File: src/app/api/invites/send/route.ts

    The invite route checks for duplicate members and pending invites, but never
    checks whether the org has already hit its member limit (Free: 2, Pro: 10,
    Team: unlimited). A free user can invite 50 people with no API-level block.

    Fix: Before creating the invite, count current members + pending unexpired
    invites. Reject if >= plan limit.

---

C3. "View Demo" button is a dead broken link
    File: landing/src/components/hero.tsx:64

        <a href="#demo">View Demo</a>

    There is NO element with id="demo" anywhere on the landing page. Clicking
    this button does nothing. This is one of the two hero CTAs and makes the
    product look broken to every first-time visitor.

    Fix options:
      (a) Remove the button until a demo video exists, OR
      (b) Link directly to a Loom/YouTube URL, OR
      (c) Open a modal
    The memory notes "Record Loom demo" as a pending manual task — that task
    must be completed before this button is live.

---

C4. "First 200 users get Pro free for 3 months" — unenforceable promise
    File: landing/src/components/signup-form.tsx:52

    The signup form displays:
      "First 200 users get Pro free for 3 months — no credit card required."

    This promise:
      - Is NOT tracked anywhere (no counter, no DB field)
      - Is NOT enforced anywhere in the app or Stripe
      - Has NO mechanism to grant free Pro access
      - Conflicts with the project note: "No waitlist concept — direct sign-up,
        all CTAs say Start Free / Get Started"

    Users who sign up expecting free Pro and don't receive it will feel
    deceived. This is the single most likely source of early trust damage and
    potential disputes.

    Fix: Either implement it (Stripe coupon + signup counter) or remove it
    entirely and replace with accurate copy.

---

C5. Billing UI says "All prices in USD" — prices are AUD
    File: src/app/(dashboard)/dashboard/billing/billing-client.tsx:272

        "Payments powered by Stripe. Cancel anytime. All prices in USD."

    The Stripe configuration and project memory clearly state AUD pricing
    ($29 AUD/mo, $99 AUD/mo). This is factually wrong. Australian users will
    see "$29 USD" charged and think something is wrong. Non-AU users get charged
    AUD (cheaper) without knowing — creating confusion either way.

    Fix: Change to "All prices in AUD" or dynamically show the currency.


HIGH PRIORITY ISSUES — Fix before significant user volume
==========================================================

H1. No brute-force protection on PIN verification
    File: src/app/api/auth/verify-pin/route.ts

    The send-PIN route rate limits to one request per 60 seconds. The verify-PIN
    route has NO attempt limit. A 6-digit PIN (1,000,000 combinations) expires
    in 10 minutes. An attacker who knows a target's email can send a PIN then
    hammer verify with all possible values. bcrypt at cost 10 (~100ms/compare)
    limits to ~6,000 attempts in 10 minutes — not sufficient for a targeted
    attack.

    Fix: Track failed attempts. After 5 failed attempts, mark the PIN invalid
    and require the user to request a new one.

---

H2. HTML injection in alert and invite emails
    Files: src/lib/alert-engine.ts, src/lib/email.ts

    User-controlled strings are interpolated directly into email HTML without
    any escaping:

      alert-engine.ts:
        <h1>${alert.name}</h1>
        ${wfRows}  (contains workflow names)

      email.ts (invite):
        ${inviterName} has invited you to join ${orgName}

    A malicious user who creates an alert named:
        <img src=x onerror="...">
    or an org named with HTML markup will have it rendered in recipient email
    clients. Most clients strip scripts but not all tags — layout can break or
    spoofed UI elements can appear.

    Fix: Add a minimal HTML escape helper and apply it to all user-supplied
    strings before interpolating into HTML email templates.

---

H3. Cron sync route has stale/misleading Vercel comments
    File: src/app/api/cron/sync/route.ts:5-9

        // GET /api/cron/sync — Vercel Cron, runs every 5 minutes
        // Vercel sets Authorization: Bearer <CRON_SECRET> automatically
        // In production, Vercel injects CRON_SECRET automatically.

    The app is deployed on Railway, using cron-job.org — not Vercel. The auth
    logic works correctly (validates the Authorization header), but the wrong
    comment implies there is automatic Vercel injection and could mislead future
    developers into a misconfiguration.

    Fix: Update comments to reflect Railway + cron-job.org.

---

H4. Landing railway.json missing explicit HOST binding
    File: landing/railway.json

        "startCommand": "npm run start"

    This maps to plain "next start" with no -H 0.0.0.0 or port flag. The main
    app correctly uses:
        "next start -H 0.0.0.0 -p ${PORT:-3000}"

    Next.js 13+ reads PORT env automatically and typically binds to 0.0.0.0 in
    production, so this likely works. But it is inconsistent with the main app
    and relies on Next.js internals rather than explicit config. If Railway
    changes how it routes or Next.js changes defaults, the landing breaks
    silently with no obvious cause.

    Fix: Update landing/package.json start script to:
        "next start -H 0.0.0.0 -p ${PORT:-3000}"

---

H5. Status page uses self-fetch with localhost fallback
    File: src/app/status/[slug]/page.tsx:29

        const baseUrl = process.env.NEXT_PUBLIC_APP_URL ?? "http://localhost:3000";
        const res = await fetch(`${baseUrl}/api/status/${slug}`, ...);

    The public status page makes an HTTP request from the server component back
    to itself. If NEXT_PUBLIC_APP_URL is ever unset or misconfigured, all status
    pages silently break (the server tries to reach localhost:3000 on Railway,
    which won't respond). The status page is customer-facing — agencies share
    it with clients — and should never have a silent failure path.

    Additionally this is a redundant network round-trip on every page load.

    Fix: Extract the DB query logic from /api/status/[slug]/route.ts into a
    shared function and call it directly from the page component.

---

H6. Landing testimonials are entirely fabricated
    File: landing/src/components/trust.tsx

    All 5 testimonials (Marcus T., Priya S., Tom W., Jess R., Daniel K.) are
    placeholder content written for a product that has not yet launched. The
    project memory notes this as a known manual task to replace before launch,
    but it is currently not done.

    Shipping fake testimonials risks credibility if any early user, journalist,
    or reviewer notices. No search results exist for these names + FlowMonix.

    Fix: Collect real beta/early-user feedback before launching, OR replace the
    testimonials section with a "What you can expect" benefits list that makes
    no fabricated claims.

---

H7. "Join 100+ n8n users already signed up" — verify before launch
    File: landing/src/components/hero.tsx:73

    This micro-proof line should be verified against the actual count in the
    signups table before going live. If the real number is under 100, this is
    a false claim. If it is over 200, the number is outdated.

    Fix: Check the Supabase signups table count. Update the copy to match
    reality, or remove it entirely.


NICE-TO-HAVE IMPROVEMENTS
===========================

N1. Sync fetches only 250 most recent executions
    File: src/lib/sync.ts:38
    Heavy n8n users with many workflows could have >250 executions in a 5-minute
    window. Executions beyond the limit would be silently missed, causing gaps
    in incident detection. Low risk at launch, grows over time.

N2. vercel.json is an empty file at project root
    File: vercel.json
    Content: {}
    Serves no purpose since the app is on Railway. Harmless but confusing for
    future contributors.

N3. Error detail fetching fires 50 concurrent requests per instance
    File: src/lib/sync.ts:55-70
    Promise.all over all failed executions with no concurrency limit. An instance
    with 50+ failures would fire 50 simultaneous HTTP requests to n8n, each with
    a 10s timeout. Could cause the cron to run long and hit Railway's request
    timeout on bad instances.

N4. "Early Access" section label is contradictory
    File: landing/src/components/signup-form.tsx:47
    The section header says "Early Access" but the product is live with direct
    signup available to anyone. This framing implies limited or exclusive access
    that does not match reality. Recommend updating to "Get Started" or "Create
    Your Account."

N5. Free plan AI uses openrouter/free auto-router
    File: src/lib/ai-debug.ts:56
    The openrouter/free pseudo-model routes to whatever free model is currently
    available. Model quality, response format, and availability can change
    without notice, silently degrading the free-tier AI experience for all users.

N6. Incident auto-resolve window is hardcoded at 30 minutes
    File: src/lib/incident-grouping.ts:37
    Not configurable per org or plan. A reasonable default, but worth noting for
    future roadmap.

N7. Delete instance is a hard delete, not soft delete
    File: src/app/api/instances/[id]/route.ts — DELETE handler
    The comment says "soft delete via is_active=false, or hard delete" but the
    implementation hard-deletes. If cascade delete is set on synced_executions
    or incidents, an accidental instance deletion permanently destroys all
    associated history with no recovery path.

BOTTOM LINE
============

Five items must be fixed before any real user sees the product:

  C1 + C2 — Billing bypass vulnerabilities. Free users can exceed plan limits
             at the API level with no enforcement.

  C3       — Broken hero CTA. "View Demo" goes nowhere. Looks unfinished.

  C4       — False promotional promise. "Pro free for 3 months" is displayed
             but not tracked or fulfilled. Legal and trust risk.

  C5       — Wrong currency. "All prices in USD" when Stripe is in AUD.
             Guaranteed to cause confusion and potential disputes.

The high-priority items (H1-H7) are real risks but survivable short-term
with active monitoring. They should be addressed within the first two weeks
of launch.
